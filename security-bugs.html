BUG-001 HIGH NOLOCK dentro de string literal — falso positivo
Descripción

containsDangerousHints() detecta WITH (NOLOCK) aunque esté dentro de un string literal SQL. stripStringLiterals() no se aplica antes de este check, causando que queries válidas sean rechazadas.

❌ Bloqueado incorrectamente
SELECT 'WITH (NOLOCK) es peligrosa'
       AS info
-- Error: forbidden table hint
-- FALSO POSITIVO
✅ Debería pasar
SELECT 'WITH (NOLOCK) es peligrosa'
       AS info
-- Es un string, no un hint real
-- Debe ejecutarse sin error
Fix requerido

En la función validateQuery(), aplicar stripStringLiterals() antes de llamar a containsDangerousHints(). Actualmente solo se aplica antes del check de CHAR concatenation.

// Orden correcto de validación:
stripped := stripStringLiterals(query)
stripped  = stripAllComments(stripped)

// Todos los checks sobre 'stripped', no sobre 'query' original
containsDangerousHints(stripped)   // ← usar stripped
containsWaitfor(stripped)          // ← usar stripped
containsOpenrowset(stripped)       // ← usar stripped
BUG-002 HIGH Subquery sobre tablas no-whitelisted pasa sin bloqueo
Descripción

validateSubqueriesForRestrictedTables() no cubre tablas de sistema (sys.*, INFORMATION_SCHEMA.*). Una IA puede acceder a tablas fuera de la whitelist envolviéndolas en subqueries, con resultado vacío pero sin error. El acceso se produce.

❌ Pasa sin bloqueo
SELECT *
FROM (
  SELECT name
  FROM sys.objects   ← no en whitelist
) AS x

-- Resultado: 0 rows (BD vacía)
-- Pero el acceso ocurrió
✅ Debe bloquearse
SELECT *
FROM (
  SELECT name
  FROM sys.objects
) AS x

-- Error esperado:
-- "table 'sys.objects' not in whitelist"
Fix requerido

Ampliar validateSubqueriesForRestrictedTables() para extraer todas las referencias a tablas dentro de subqueries — incluyendo schemas (sys., dbo., INFORMATION_SCHEMA.) — y verificarlas contra la whitelist. Usar el mismo parser de tablas que se usa en la validación principal.

// Regex actual probablemente solo captura tabla simple:
FROM\s+(\w+)

// Debe capturar también schema.tabla:
FROM\s+([\w]+\.[\w]+|[\w]+)

// Y aplicar la misma whitelist check que la query principal
if !isTableWhitelisted(table) {
    return fmt.Errorf("table '%s' in subquery not in whitelist", table)
}
BEH-001 DECISIÓN SEL/**/ECT — keyword partido llega al engine SQL
Descripción

Cuando un keyword SQL se parte con comentario inline (SEL/**/ECT), el MCP no lo bloquea proactivamente — llega al engine de SQL Server que lo rechaza por sintaxis inválida. No es explotable, pero es inconsistente con la filosofía de "bloquear antes de ejecutar".

Comportamiento observado
-- Input:
SEL/**/ECT 1

-- MCP: no bloquea
-- SQL Server: "Incorrect syntax near '1'"
-- Resultado final: error (no explotable)

-- Sí bloquea keywords COMPLETOS en comentarios:
/*DROP*/SELECT 1  → "keyword hidden in comments" ✅
Opciones

Opción A (recomendada): Después de stripAllComments(), verificar que la query resultante no contiene keywords SQL partidos. Si stripAllComments("SEL/**/ECT") produce "SELECT", ese string ya pasa la validación normal — el fix es asegurar que la validación de keywords se hace siempre sobre la versión post-strip.

Opción B: Aceptar el comportamiento actual — SQL Server lo rechaza de todas formas, el riesgo real es nulo. Solo documentarlo como known behavior.
